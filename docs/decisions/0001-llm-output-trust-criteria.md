# ADR-0001: LLM出力の信頼判定基準

## 状態

採用済み

## 背景

Offline LLM Eval は、LLM出力を fixture（再現用の固定データ）に対して継続評価する。評価結果を再利用するには、どの条件を満たした出力を「合格」と扱うかを機械的に決める必要がある。

v0.1.0では、LLM judge（LLMによる評価）ではなく、応答形式、引用、回答しないべきケース、禁止語、応答時間、外部サービス相当のエラー、secret値に似た文字列パターンなどの明示的な条件を使う。

## 決定

LLM出力は、宣言された応答モードに合い、対象の判定条件が合格または許容された注意状態になり、secret値に似た文字列パターンが検出されない場合にのみ合格として扱う。

secret値に似た文字列パターンの検出失敗は、確認結果で上書きしない。検証証跡には `not_claimed`（主張しない範囲）を含め、この実行結果から主張しない範囲を明示する。

## 検討した代替案

1. HTTP 200や外部サービス相当の成功だけで合格扱いにする。応答形式違反、引用不整合、回答しないべきケースの誤答を見落とすため採用しない。
2. LLM judge（LLMによる評価）だけで合否を決める。v0.1.0では再現性と説明可能性を優先するため採用しない。
3. 確認結果で全失敗を上書き可能にする。secret値に似た文字列の検出結果を見えなくするリスクがあるため採用しない。

## トレードオフ

- [+] 合否理由を fixture（再現用の固定データ）と判定条件に紐づけて再実行できる。
- [+] secret値に似た文字列の検出結果を、ケース状態とは別に品質判定と検証証跡へ残せる。
- [-] 曖昧な品質や好みの評価はv0.1.0では扱わない。
- [-] 外部LLMサービスの実品質や費用はこの判断だけでは評価しない。

## 検証

- 評価条件のテストで、各判定種別の合格、失敗、注意、スキップを確認する。
- 実行処理のテストで、mock provider の応答、外部サービス相当のエラー、応答モード不一致、secret値に似た文字列の検出経路を確認する。
- 品質判定のテストで、secret値に似た文字列の検出失敗と重要失敗が失敗判定に残ることを確認する。
- 検証証跡のテストで、secret値に似た文字列の検出結果と `not_claimed` がMarkdownに出ることを確認する。

## 影響

- 検証証跡で主張する合格または失敗は、この信頼判定基準に基づく。
- 信頼判定基準を満たさない出力は、外部サービス相当の呼び出しが成功していても失敗または確認待ちとして扱う。
- 将来LLM judge（LLMによる評価）を導入する場合も、この基準の補助として扱う。
